# OTA 升级与 Bootloader 设计规范

固件远程升级的架构设计、Flash 分区、可靠性保障与安全要求。

## 1. 方案选型

| 方案 | 适用 | 说明 |
|------|------|------|
| ESP32 原生 OTA | ESP32 系列 | esp_https_ota + 双 app 分区，官方方案直接用 |
| 自研 Bootloader 双分区 | STM32 等裸片 | 本文重点，A/B 分区 + 引导切换 |
| 单分区 + 外部 Flash 暂存 | 片内 Flash 不足 | 新固件先存 W25Q，校验后搬运 |
| Ymodem/串口 IAP | 无网络产品 | 生产/售后线下升级 |

## 2. Flash 分区设计（STM32 双分区示例）

```
0x08000000 ┌──────────────┐
           │ Bootloader   │ 16~32KB, 永不升级, 只做校验与跳转
0x08008000 ├──────────────┤
           │ App A (运行) │ (Flash-32KB)/2
0x08042000 ├──────────────┤
           │ App B (下载) │ 同上
0x0807C000 ├──────────────┤
           │ 参数区        │ 升级标志/版本号/CRC, 独立扇区
           └──────────────┘
```

- App 编译时需改链接脚本 FLASH ORIGIN，且设置中断向量表偏移 `SCB->VTOR`
- 参数区结构：`{magic, active_slot, version, image_size, image_crc32, boot_try_cnt}`，写入用"先写副本再改标志"防掉电

## 3. Bootloader 核心流程

```
上电 → 读参数区
  ├─ 升级标志有效? → 校验 B 区 CRC32
  │     ├─ 通过 → (可选)搬运/直接标记 B 为 active → 跳转
  │     └─ 失败 → 清标志, 跳回 A 区（回滚）
  ├─ boot_try_cnt ≥ 3? → 自动回滚到上一分区
  └─ 校验 active 分区 CRC → 跳转 App
App 启动成功后: 清零 boot_try_cnt（自确认机制）
```

跳转前必须：关闭全部中断与外设时钟、复位 SysTick、设置 MSP，再跳 Reset_Handler。

## 4. 传输与可靠性

- **分片传输**：每片 512~4KB，带片序号 + 片 CRC；支持断点续传（记录已写偏移）
- **整包校验**：全部写完后对镜像算 CRC32/SHA256 与服务器下发值比对，不通过不置升级标志
- **掉电安全**：任何时刻掉电都必须能启动——下载区写坏不影响运行区；参数区双副本
- **版本防降级**：参数区记录版本号，低版本镜像拒绝（除非强制标志）
- **失败回滚**：新固件启动后 N 秒内未自确认（喂狗+置标志）则 bootloader 回滚

## 5. 安全要求

- 固件镜像签名验证（RSA2048/ECDSA-P256），公钥固化在 bootloader；至少做 SHA256 + 密钥 HMAC
- 传输走 HTTPS/TLS 或私有协议加密，禁止明文 HTTP 裸传生产固件
- 关闭调试口读保护（STM32 RDP Level 1+）防止镜像被读出
- bootloader 自身不可被 OTA 覆盖；升级 bootloader 需专门的两级引导方案

## 6. 测试清单

- [ ] 正常升级 100 次成功率 100%

- [ ] 写参数区瞬间断电，设备能启动
- [ ] 刷入 CRC 错误/签名错误镜像被拒绝
